# Listener Architecture V1 与 macOS System Listener Prototype

## 背景

Companion Runtime 已具备 Internal Event、Behavior Slot、Action Resolver、Character Pack 和统一 Composition Root，但缺少对外部世界进行感知的独立边界。如果由平台代码直接调用 Runtime、PetManager 或 Viewer，会使平台事实、行为决策和角色表现重新耦合。

本阶段增加 Listener Layer：Listener 只观察外部事实并输出 `ExternalEvent`，随后由独立 Mapping 转换为 Internal Event。Runtime Core 不感知 Listener 或 macOS。

## 架构

```text
macOS System
    ↓ sample
MacSystemListener
    ↓ ExternalEvent
ListenerManager / Desktop IPC
    ↓
ExternalEventMapper
    ↓ Raw Internal Event
EventNormalizer
    ↓ CompanionEvent
runtime.publish()
    ↓
Behavior Slot → Current Character → Action → Asset
```

职责边界：

- `Listener`：定义 `id`、`start()`、`stop()` 和 `onEvent()`。
- `ListenerManager`：注册 Listener，并统一启动、停止；启动失败时停止已经启动的 Listener。
- `ExternalEvent`：只描述外部来源、名称、时间和 payload，不包含 Internal Event 的 `type`，也不包含角色、Behavior、Action 或 Asset。
- `ExternalEventMapper`：根据 `source:name` 显式转换为 Internal Event 输入；不存在 Mapping 时明确报错。
- Desktop：作为宿主注册真实 Listener，通过安全 IPC 把 External Event 送至 Renderer，再调用现有 `EventNormalizer` 和 `runtime.publish()`。

ListenerManager 不依赖 Pet、Behavior、Action 或 Viewer。Core Runtime 没有为 Listener 增加特殊入口。

## 当前实现

新增目录：

```text
packages/listeners/
├── core/
│   ├── ExternalEvent.ts
│   ├── ExternalEventMapper.ts
│   ├── Listener.ts
│   └── ListenerManager.ts
├── system/
│   ├── SystemListener.ts
│   └── macos/MacSystemListener.ts
└── tests/listener.test.ts
```

`MacSystemListener` 当前每 5 秒采样：

- CPU 使用率：根据两次 `node:os.cpus()` 累积时间差计算，默认阈值 90%。
- Memory 使用率：根据 `totalmem()` 与 `freemem()` 计算近似使用率，默认阈值 90%。这是 V1 轻量指标，不等价于 macOS 原生 memory pressure level。

Listener 只在指标从阈值下方进入阈值上方时发出 `cpu_high` 或 `memory_pressure`，避免持续高负载时重复发布。指标恢复后再次越界可以重新触发。

Desktop 宿主配置以下显式映射：

```text
system:cpu_high
  → CUSTOM_EVENT / CPU_HIGH
  → EXECUTING

system:memory_pressure
  → CUSTOM_EVENT / MEMORY_PRESSURE
  → ERROR
```

前半段由 `ExternalEventMapper` 完成；后半段作为 Desktop 注入的 Event Mapping 和 Behavior Rule 完成。`MacSystemListener` 不知道这些 Behavior Slot，也不参与当前角色或 Action 选择。

## 设计原因

- External Event 与 Internal Event 使用不同类型，防止平台监听代码伪装成 Runtime 业务事件。
- 指标提供器可注入，使真实 `MacSystemListener` 能用确定性指标测试，无需制造机器高负载。
- Listener 在 Electron Main Process 运行，Renderer 保持 Sandbox；External Event 通过窄 IPC Bridge 传递。
- Mapping 留在宿主集成边界，因此未来不同宿主可以对同一外部事实采用不同配置，而无需修改 Listener。

## 风险

- `freemem()` 只能提供轻量内存使用率近似，不能完整表示 macOS 原生内存压力、压缩内存和 swap 状态。
- V1 尚未提供 Listener handler 的单独取消订阅接口；Listener 生命周期与宿主进程一致。
- Desktop 当前使用代码内显式 Mapping。未来配置化时必须进行 schema 校验，不应让 Listener 自己读取 Runtime 配置。
- CPU/Memory 默认阈值需要通过真实长期运行数据校准，当前值只用于原型。

## 未实现

本阶段明确不包含：

- Windows System Listener
- Git Listener
- Codex Listener
- Application Listener
- User Listener 插件加载
- 第三方插件或 Marketplace
- Disk Listener
- macOS 原生 memory pressure API

## 验证结果

2026-07-18 执行：

- `npm run typecheck`：通过，strict 模式无 `any`。
- `npm test`：通过，39/39；原有 35 个测试继续通过，新增 4 个 Listener 测试通过。
- `npm run desktop:build`：通过。
- Listener Interface：验证 start、stop 和事件回调。
- ListenerManager：验证 register、startAll、stopAll。
- External Event：验证输出不存在 Internal Event `type`。
- 完整集成：注入高内存指标后，真实 `MacSystemListener` 输出 `memory_pressure`，经 Mapping、Normalizer、Runtime、Behavior Engine 后，由当前 Sasuke 解析为 `danger` Action。
- macOS Desktop 实际启动：ListenerManager 和 MacSystemListener 正常初始化，窗口显示 `READY: IDLE / idle`，无启动错误。真实机器当次指标未越过阈值，因此没有把未发生的系统告警记录为实机事件。

## 结果

Listener Architecture V1 已完成。Companion Runtime 现在具备平台无关 Listener 契约、统一生命周期、独立 External Event 模型、第一个 macOS System Listener，以及不修改 Runtime Core 的完整事件闭环。
